============================================================
WHAT IS GENUINELY STRONG
============================================================
1) Real architecture exists (not random React soup)
- There is a deliberate orchestration model across:
  - timeline state (lib/gsapTimeline.ts)
  - central director store (lib/useDirector.ts)
  - runtime scene control (lib/SceneDirector.ts)
  - asset/frame orchestration (lib/AssetOrchestrator.ts)

2) Performance-conscious design is real
- Tiered behavior and quality degradation paths are built in.
- Progressive scene phase loading exists in scene groups.
- Frequent use of refs/subscriptions avoids unnecessary React render pressure in frame loops.

3) Scene engineering depth is above average
- Complex lighting/shader setups, cross-scene transitions, and cinematic state handling are non-trivial and intentionally designed.
- This is not beginner-level architecture.

4) Organized folder decomposition
- app/, components/, hooks/, lib/ are logically separated and domain-meaningful.

============================================================
WHERE IT BREAKS DOWN
============================================================
1) Monolith risk is high
- Several files are too large and carry mixed responsibilities (rendering + state + debug + tuning + UI controls + orchestration in one module).
- This creates change-risk and regression-risk.

2) Debug/prototype residue leaks into production paths
- High volume of console logging and manual debug plumbing remains.
- "Locked values / do not modify" comments appear repeatedly, which usually indicates insufficient abstraction/testing around tuning systems.

3) Type safety is inconsistent in critical areas
- Heavy use of any in key runtime/rendering modules.
- This weakens static guarantees exactly where you need them most.

4) UI maintainability debt
- Very heavy inline style usage and repeated panel/menu patterns.
- Overlay/debug UIs are powerful but difficult to evolve consistently.

5) Imperative browser control is extensive
- Frequent direct DOM style writes, timers, and global window coupling.
- Works, but increases fragility and race-condition surface area.

6) Testing gap
- No tests in src tree.
- For a system this stateful and timing-sensitive, that is a major long-term risk.

7) Encoding quality issue
- Mojibake/garbled characters appear in comments/strings in many files, indicating encoding inconsistencies.
